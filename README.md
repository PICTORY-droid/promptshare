# 수업 커리큘럼 반영

PromptLab v3는 “AI Vibe Coding: 웹 서비스 개발2” 수업에서 배운 내용을 실제 웹 서비스 개발 과정에 반영한 프로젝트입니다.

단순히 기능을 나열하는 방식이 아니라, 수업에서 배운 개념을 PromptLab의 기획, 기능 설계, 보안 정책, 데이터 구조, 배포 과정, 시연 흐름에 연결했습니다.

## 왜 만들었나

AI 도구가 많아질수록 프롬프트를 잘 쓰는 능력이 중요해졌습니다.  
하지만 실제 사용자는 좋은 프롬프트를 어디에 저장하고, 어떻게 다시 쓰고, 공개 전에 어떤 위험을 확인해야 하는지 알기 어렵습니다.

처음 PromptLab은 수업 초기에 만든 프롬프트 공유형 학습 플랫폼이었습니다.  
당시에는 카테고리별 프롬프트를 많이 모아 보여주고, 사용자가 여러 예시를 보면서 프롬프트 작성 방법을 익히는 구조를 목표로 했습니다.

실제로 기존 버전에서는 약 900개 이상의 프롬프트 데이터를 모아 운영하는 방향까지 만들어 보았습니다.  
이 시기에는 프롬프트를 많이 확보하고, 카테고리별로 탐색할 수 있게 만드는 것이 핵심이라고 생각했습니다.

하지만 수업을 계속 들으면서 Vibe Coding, Prompt, Context, PRD, Customer Definition, User Story, LLM, AI Agent, Context Engineering, MCP, Rule, Supabase, CRUD, Authentication 같은 내용을 배우게 되었습니다.  
그 내용을 프로젝트에 조금씩 반영하다 보니, 단순히 프롬프트를 많이 보여주는 구조만으로는 부족하다는 점을 알게 되었습니다.

프롬프트는 개수가 많은 것보다 사용 목적이 분명하고, 예시가 이해하기 쉬우며, 실제 업무나 창작 과정에서 안전하게 재사용할 수 있어야 합니다.  
또한 프롬프트에는 개인정보, 회사기밀, 계약조건, 저작권 위험, 허위·과장 표현이 포함될 수 있기 때문에 저장과 공개 전에 안전 검사가 필요하다고 판단했습니다.

그래서 PromptLab v3에서는 기존처럼 프롬프트 수를 늘리는 방향보다, 사용자가 직접 좋은 프롬프트를 작성하고, 저장하고, 수정하고, 공개 여부를 관리할 수 있는 구조로 업데이트했습니다.

또한 별도로 진행하던 AI SafeCheck 프로젝트의 방향을 PromptLab 내부 기능으로 통합했습니다.  
프롬프트를 저장하거나 공개하기 전에 위험 요소를 검사하고, 필요한 경우 안전한 수정 방향을 안내하는 구조를 만들었습니다.

PromptLab v3는 단순한 프롬프트 공유 사이트가 아니라, 수업을 들으며 배운 웹 서비스 개발 흐름을 반영해 다시 설계한 안전한 AI 프롬프트 작업실입니다.

### 1. Vibe Coding

PromptLab v3는 Vibe Coding 방식으로 개발했습니다.

처음부터 완성된 큰 기능을 한 번에 만들지 않고, 기능을 작은 단위로 나누어 구현했습니다.

예를 들면 다음 순서로 작업했습니다.

- 프로젝트 v3 방향 재정의
- Supabase Auth 연결
- 프롬프트 CRUD 구현
- AI SafeCheck 검사 엔진 구현
- 저장 전 SafeCheck 정책 연결
- SafeCheck 리포트 저장
- Google, Kakao 로그인 추가
- 반응형 화면 보완
- MCP-style tool adapter 추가
- README와 Demo Day 흐름 정리

각 단계는 기능 단위로 Git 커밋을 남기고, TypeScript 검사, 테스트, 빌드 검증을 거쳐 진행했습니다.

이 과정에서 단순히 코드를 만드는 것이 아니라, 구현 후 실제 화면에서 사용자 입장에서 어색한 부분을 다시 수정했습니다.

예를 들면 버튼 터치 반응, 작성 화면 안내 문구, 모바일 메뉴, 대시보드 카드, 리포트 카드, 프롬프트 수정 화면을 반복적으로 다듬었습니다.

### 2. Prompt

PromptLab의 핵심 데이터는 프롬프트입니다.

프롬프트를 단순한 텍스트 한 덩어리로 저장하지 않고, 사용자가 나중에 다시 이해하고 재사용할 수 있도록 구조화했습니다.

현재 프롬프트 작성 화면은 다음 항목으로 구성되어 있습니다.

- 제목
- 카테고리
- 사용 목적
- 프롬프트 본문
- 예시 입력
- 예시 출력
- 안전 주의사항
- 공개 범위
- 상태

프롬프트 본문에는 AI에게 실제로 입력할 지시문을 작성합니다.

예시 입력은 사용자가 이 프롬프트에 넣을 수 있는 입력값을 보여주는 항목입니다.

예시 출력은 AI가 어떤 형식으로 답해야 하는지 보여주는 샘플입니다.

안전 주의사항은 개인정보, 회사기밀, 저작권 위험, 허위·과장 표현처럼 프롬프트 사용 시 조심해야 할 내용을 기록하는 항목입니다.

이렇게 프롬프트를 구조화하면 단순 저장보다 재사용성과 관리성이 높아집니다.

### 3. Context

PromptLab v3는 사용자별 컨텍스트를 분리합니다.

로그인한 사용자는 본인이 작성한 프롬프트만 대시보드에서 관리할 수 있습니다.

공개 프롬프트 목록에는 공개 + 게시 상태인 프롬프트만 표시됩니다.

비공개 + 초안 프롬프트는 개인 작업용으로 본인 대시보드에서만 볼 수 있습니다.

SafeCheck 리포트도 로그인한 사용자 기준으로 저장되고 조회됩니다.

즉, 같은 서비스 안에서도 다음 컨텍스트가 분리되어 있습니다.

- 개인 작업 공간
- 공개 프롬프트 목록
- SafeCheck 검사 결과
- 관리자 정책 안내
- 로그인 세션

이 구조를 통해 사용자는 테스트 중인 프롬프트를 공개하지 않고 안전하게 보관할 수 있습니다.

### 4. PRD

프로젝트는 `docs/PRD.md`를 기준으로 방향을 잡았습니다.

PRD에서는 PromptLab v3의 목표를 단순 프롬프트 공유 플랫폼이 아니라, 안전한 프롬프트 작성과 관리가 가능한 서비스로 정의했습니다.

v3에서 중요한 목표는 다음과 같습니다.

- 기존 900여 개 저품질 프롬프트를 핵심 데이터로 유지하지 않기
- 사용자별 로그인 기반 프롬프트 관리 구현
- 프롬프트 작성, 수정, 공개, 비공개, 보관, 복구 흐름 구현
- AI SafeCheck를 저장 전 안전 검사 모듈로 통합
- 검사 원문을 저장하지 않는 리포트 구조 구현
- 수업 제출과 실제 운영이 모두 가능한 구조 만들기

PRD를 기준으로 기능 범위를 정하고, MVP에서 제외할 기능도 명확히 했습니다.

MVP에서 제외한 기능은 다음과 같습니다.

- 유료 LLM API 필수 의존
- PDF, HWP, OCR 업로드
- 대용량 문서 분석
- 기업 SSO
- 네이버 로그인
- 모바일 네이티브 앱 직접 개발

이렇게 범위를 제한해 프로젝트가 커지면서 불안정해지는 것을 막았습니다.

### 5. Customer Definition

PromptLab v3의 주요 사용자는 프롬프트를 안전하게 작성하고 관리하려는 사람입니다.

대표 사용자는 다음과 같습니다.

- AI 프롬프트 작성이 익숙하지 않은 초보 사용자
- 업무용 프롬프트를 정리하고 싶은 실무자
- 공개 전에 개인정보나 회사기밀 위험을 확인하고 싶은 사용자
- 수업 과제로 실제 동작하는 웹 서비스를 제출해야 하는 학습자
- 추후 모바일 앱으로도 사용할 수 있는 프롬프트 작업 공간이 필요한 사용자

이 사용자들은 단순히 많은 프롬프트를 보는 것보다, 내가 만든 프롬프트를 안전하게 저장하고 다시 활용하는 흐름이 더 중요합니다.

그래서 PromptLab v3는 대량 프롬프트 목록보다 개인 작성, SafeCheck, 리포트, 공개 관리에 집중했습니다.

### 6. User Story

PromptLab v3의 핵심 사용자 흐름은 다음과 같습니다.

사용자는 Google 또는 Kakao로 로그인합니다.

로그인 후 `/write` 화면에서 새 프롬프트를 작성합니다.

작성한 프롬프트 본문을 저장 전 AI SafeCheck로 검사합니다.

검사 결과가 allow이면 저장할 수 있습니다.

review이면 비공개 + 초안으로만 저장할 수 있습니다.

block이면 저장이 차단됩니다.

저장한 프롬프트는 `/dashboard`에서 확인할 수 있습니다.

사용자는 대시보드에서 프롬프트를 상세 확인하거나 수정하거나 보관할 수 있습니다.

공개할 준비가 끝난 프롬프트는 공개 + 게시 상태로 바꿀 수 있습니다.

공개 + 게시 상태의 프롬프트는 `/prompts` 목록에 표시됩니다.

SafeCheck 검사 결과는 `/reports` 화면에서 확인할 수 있습니다.

이때 검사 원문은 저장되지 않고, 점수와 판정, 위험 카테고리, 안전 문장 안내만 저장됩니다.

### 7. LLM

수업에서 LLM 개념을 배웠지만, PromptLab v3의 MVP에서는 실제 LLM API를 사용하지 않았습니다.

그 이유는 명확합니다.

PromptLab은 사용자가 작성한 프롬프트 안에 개인정보, 회사기밀, 계약조건, 내부자료가 들어갈 가능성이 있습니다.

이 원문을 외부 LLM으로 보내면 보안 위험이 생길 수 있습니다.

또한 LLM은 같은 입력에도 출력이 달라질 수 있기 때문에, 저장 차단이나 위험 판정처럼 일관성이 필요한 기능에는 적합하지 않을 수 있습니다.

그래서 PromptLab v3의 SafeCheck는 룰 기반 검사로 구현했습니다.

개인정보, 회사기밀, 계약정보, 저작권 위험, 허위·과장 표현을 정해진 룰과 점수 기준으로 탐지합니다.

향후 LLM을 붙인다면 위험 판정이 아니라, 마스킹된 문장을 더 자연스럽게 고쳐주는 선택 기능으로만 붙이는 것이 안전합니다.

### 8. AI Agent

AI Agent 개념은 두 가지 방식으로 반영했습니다.

첫 번째는 개발 과정입니다.

PromptLab v3는 Antigravity 환경에서 Claude CLI와 Gemini CLI를 활용해 개발했습니다.

기능을 작은 작업으로 나누고, 각 단계마다 코드를 작성하고, 오류를 확인하고, 테스트와 빌드를 실행하고, GitHub에 push하는 흐름을 반복했습니다.

기존 PromptLab도 Claude CLI를 활용해 만들고 자동 업로드 흐름을 사용했습니다.

두 번째는 서비스 기능입니다.

PromptLab 내부에서 SafeCheck는 사용자의 프롬프트를 검사하고, 위험을 판정하고, 수정 방향을 안내하는 보조 검사 에이전트 역할을 합니다.

다만 현재 PromptLab이 완전한 자율 AI Agent를 구현했다고 표현하지는 않습니다.

현재 구현은 안전 검사 중심의 보조 에이전트 구조입니다.

### 9. Context Engineering

PromptLab v3는 Context Engineering 관점에서 원문을 무조건 저장하지 않습니다.

특히 SafeCheck 검사 원문은 저장하지 않습니다.

대신 리포트에는 서비스 운영에 필요한 최소 정보만 저장합니다.

저장하는 정보는 다음과 같습니다.

- 위험 점수
- 최종 판정
- 위험 카테고리
- 안전 문장 안내
- 정책 버전
- 탐지기 버전
- 검사 시각

저장하지 않는 정보는 다음과 같습니다.

- 검사 원문 프롬프트
- 고객명
- 전화번호
- 이메일 원문
- 회사기밀 원문
- 내부자료 원문
- 계약조건 원문
- 상담기록, 진료기록 같은 민감한 원문

이 구조는 사용자의 입력을 최대한 덜 보관하면서도, 리포트 기능은 유지하기 위한 설계입니다.

### 10. MCP-style Tool Adapter

PromptLab v3에는 실제 외부 MCP 서버를 운영하지는 않습니다.

대신 MCP와 tool-calling 개념을 코드 구조에 반영하기 위해 `features/tools` 아래에 MCP-style tool adapter를 만들었습니다.

구현된 도구는 `scan_ai_input`입니다.

이 도구는 입력된 프롬프트를 SafeCheck 룰 기반 엔진으로 검사하고 다음 정보를 반환합니다.

- 요약
- 위험 점수
- 최종 판정
- 위험 카테고리
- 탐지 개수
- 안전 문장 안내
- 정책 버전
- 탐지기 버전
- LLM 사용 여부
- 원문 저장 여부

관련 파일은 다음과 같습니다.

```text
features/tools/types.ts
features/tools/server/scan-ai-input.tool.ts
features/tools/server/tool-registry.ts
features/tools/tests/tool-registry.test.ts
