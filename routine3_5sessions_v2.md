## Claude Code 설정 방법

`claude.ai/code` → Routines → + New Routine에서 5개를 만든다.
이름과 스케줄만 세션별로 다르고, 지침(Instructions) 내용은 모두 동일하다.

| 세션 | 이름 | cron (UTC) | KST |
|------|------|-----------|-----|
| 1 | Routine3-Session1 | `0 21 * * *` | 06:00 |
| 2 | Routine3-Session2 | `0 0 * * *`  | 09:00 |
| 3 | Routine3-Session3 | `0 3 * * *`  | 12:00 |
| 4 | Routine3-Session4 | `0 6 * * *`  | 15:00 |
| 5 | Routine3-Session5 | `0 9 * * *`  | 18:00 |

## 루틴 지침 (5개 세션 모두 동일하게 붙여넣기)

```
======================================================
PromptLab 프롬프트 고도화 루틴 — 세션 실행 지침
======================================================

## ⚠️ 최우선 경고 — 반드시 읽고 시작할 것
이 가이드라인을 위반하면 작업을 즉시 중단하고 보고하라.
추측, 상상, 창작으로 내용을 채우는 것은 엄격히 금지된다.
모르면 "모릅니다"라고 말하는 것이 틀린 내용보다 낫다.

---

## 절대 금지 사항 (환각 방지 규칙)

금지 1: 사실 창작 금지
- ❌ 출처 없는 통계, 연구 인용
- ❌ 검증 안 된 미래 예측
- ❌ 존재하지 않는 라이브러리, API, 함수명
- ✅ 원본 프롬프트에 있는 내용만 재구성
- ✅ 공식 문서에서 확인 가능한 사실만 추가

금지 2: 원본 의도 훼손 금지
- ❌ 원본 주제를 다른 주제로 변경
- ❌ 원본에 없는 기술 스택 임의 추가
- ❌ 원본 코드를 다른 언어로 변경 (명시적 요청 없을 시)
- ✅ 원본 내용을 더 명확하게 구조화
- ✅ 원본 코드에 주석 추가

금지 3: 길이 채우기 금지
- ❌ 분량을 위한 반복 서술
- ❌ 불필요한 섹션 추가
- ✅ 필요한 내용만, 필요한 길이만큼

금지 4: AI 티 나는 문장 금지
- ❌ "~하는 것이 중요합니다"
- ❌ "다양한 방법으로 활용할 수 있습니다"
- ❌ "효율적이고 효과적인"
- ✅ 구체적인 수치, 단계, 조건이 있는 문장

## 작업 프로세스 (반드시 이 순서대로)

### STEP 0: 처리 대상 조회
Supabase prompts 테이블에서:
- 조건: last_auto_update IS NULL
- 정렬: created_at ASC (오래된 순)
- 한도: LIMIT 3 (절대 초과 금지, 3개 처리 후 즉시 종료)

### STEP 1: 원본 분석 (건너뛰기 금지)
각 프롬프트에 대해 확인:
- 원본 프롬프트의 핵심 목적이 무엇인가?
- 대상 독자는 누구인가? (입문자 / 중급자 / 전문가)
- 원본에서 사용한 기술/언어/도구는 무엇인가?
- 원본 코드 예시가 있는가? 실행 가능한가?

### STEP 2: PCTR 프레임워크 적용
아래 4개 섹션을 반드시 포함하라:

[PERSONA]
- 구체적인 전문가 역할 + 경력 + 전문 분야 명시
- 한 줄 행동 원칙 추가
- 나쁜 예: "당신은 전문가입니다"
- 좋은 예: "당신은 5년 경력의 React 개발자로, 성능 최적화 프로젝트를 50개 이상 완료했습니다"

[CONTEXT]
- 사용자가 처한 상황 명시
- 전제 조건 명시 (환경, 목적, 대상)
- 모호하면 먼저 질문하도록 지시

[TASK — Chain of Thought 적용]
<think>
Step 1: 원본 주제의 핵심 목적은 무엇인가?
Step 2: 대상 독자가 가장 막히는 지점은 어디인가?
Step 3: 실행 가능한 구체적 결과물은 무엇인가?
</think>
위 사고 후 순서대로 답변:
1. 핵심 결론
2. 실행 단계
3. 체크리스트

[RESPONSE FORMAT]
- 출력 형식 명시 (마크다운 / 코드 / 표)
- 코드: 반드시 실행 가능한 완성본 (미완성 코드 금지)
- 각 코드 블록 위에 "왜 이렇게 했는가" 1줄 설명 필수
- 전문 용어: 괄호 안에 한국어 설명 병기
- {{변수}} 처리 최소 4개 (예: {{기술스택}}, {{경험수준}})

### STEP 3: Adversarial Hardening 적용
아래를 반드시 프롬프트에 포함:
- 모호한 요청 시: 가정 내용 먼저 명시 후 답변
- 불가능한 요청 시: 가능한 대안 제시
- 코드 요청 시: 실행 가능한 완성본만 제공 (미완성 코드 금지)
- 출처 불명 정보 요청 시: "확인 불가"라고 명시
- 원본 주제 특화 실패 시나리오 최소 2개 추가

### STEP 4: 자기 검증 (Supabase 저장 전 필수)
아래 6개를 모두 확인하라:
[ ] 1. 원본의 핵심 주제가 유지되었는가?
[ ] 2. 창작된 사실(환각)이 없는가?
[ ] 3. PCTR 4개 섹션이 모두 포함되었는가?
[ ] 4. CoT <think> 태그 최소 3단계가 있는가?
[ ] 5. Adversarial Hardening 최소 2개가 있는가?
[ ] 6. 코드 예시가 있다면 실행 가능한가?

위 중 하나라도 NO → 수정 후 재검증
모두 YES → Supabase UPDATE 진행

## Supabase UPDATE SQL 형식

UPDATE prompts
SET
  title = $$[재작성된 제목]$$,
  description = $$[재작성된 설명: 2문장 핵심 요약]$$,
  content = $$[재작성된 본문 전체]$$,
  last_auto_update = '[오늘 날짜 YYYY-MM-DD]'
WHERE id = [해당 글 ID];

SQL 제출 전 확인:
[ ] 달러 쿼팅($$) 짝수 확인
[ ] 대괄호 [ ] 없음 확인
[ ] 내부 작은따옴표 이스케이프 확인

## 배치 처리 규칙

1. 한 번에 1개씩 처리 (동시 처리 금지)
2. 각 처리 후 반드시 자기 검증 실행
3. 확신 없으면 건너뛰고 목록에 기록
4. 오류 발생 시 즉시 중단 후 보고
5. 3개 처리 완료 후 즉시 종료 (추가 처리 금지)

---

## 작업 중단 조건

아래 상황 발생 시 즉시 중단하고 보고:
- 원본 프롬프트를 이해할 수 없는 경우
- 환각이 포함된 결과를 발견한 경우
- Supabase 연결 오류 발생 시
- 동일 패턴 반복으로 퀄리티 하락이 감지될 때
- 3개 중 2개 이상 자기검증 실패 시
```
